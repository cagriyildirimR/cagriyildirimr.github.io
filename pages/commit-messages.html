Writing Good Commit Messages

Introduction

Git commit messages are an important aspect of your version control workflow, as they allow you to communicate the changes you have made to your codebase and provide context for why those changes were necessary. Here are some key aspects of writing good Git commit messages:

  • Keep the subject line short and concise: The subject line should be no longer than 50 characters, and should clearly and concisely summarize the changes made in the commit.
  • Use the imperative mood: The subject line should be written in the imperative mood, as if you are issuing a command.
  • Capitalize the subject line: Capitalize the first letter of the subject line to make it more readable.
  • Use the body to provide more context: The body should provide more context and detail about the changes.
  • Keep it concise: Try to keep the commit message concise while still providing enough information.

Keep the subject line short and concise

Good Examples:

  • "Fix login form bug"
  • "Update README with installation instructions"

Bad Examples:

  • "I fixed the really annoying bug in the login form that was causing problems for everyone..."
  • "Updating the readme file with instructions on how to install the application..."

Use the imperative mood

Good Examples:

  • "Add tests for new feature"
  • "Refactor database schema"
  • "Fix typo in documentation"

Bad Examples:

  • "Added tests for the new feature"
  • "The database schema has been refactored"
  • "A typo was fixed in the documentation"

Summary Example

Summarize changes in around 50 characters or less

More detailed explanatory text, if necessary. Wrap it to about 72
characters or so. In some contexts, the first line is treated as the
subject of the commit and the rest of the text as the body.

Explain the problem that this commit is solving. Focus on why you
are making this change as opposed to how (the code explains that).
Are there side effects or other unintuitive consequences of this
change? Here's the place to explain them.

Further paragraphs come after blank lines.

 - Bullet points are okay, too
 - Typically a hyphen or asterisk is used for the bullet

If you use an issue tracker, put references to them at the bottom:

Resolves: #123
